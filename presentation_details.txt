# DeliKart - Project Presentation Details

## Project Overview
DeliKart is a comprehensive logistics and order-tracking portal designed to provide real-time updates and seamless management of deliveries. It features distinct interfaces for Customers, Agents, and Administrators, ensuring that cargo is tracked, managed, and delivered efficiently.

## Architecture Details

### Models (3 Core Models)
The application utilizes Eloquent ORM to interact with the MongoDB database. 
1. **User:** Manages authentication, roles (Admin, Agent, Customer), and basic user profiles.
2. **Order:** Represents a delivery manifest/cargo shipment. It contains details like sender, receiver, current status, and assigned agent.
3. **OrderStatusUpdate:** Tracks the history of an order's journey. Each time an agent updates the status, a new record is created here to provide a detailed timeline for the customer.

### Controllers (6 Core Controllers + Auth)
1. **HomeController:** Handles the routing after a successful login, redirecting users to their respective dashboards based on their role.
2. **AdminController:** Manages the administrative operations, such as assigning orders to agents, approving new agent registrations, and viewing network-wide metrics.
3. **AgentController:** Powers the agent dashboard, allowing delivery personnel to view their assigned routes and update the status of active deliveries.
4. **OrderController:** Handles the CRUD (Create, Read, Update, Delete) operations for orders, primarily utilized by admins to create or modify shipments.
5. **TrackingController:** Used by customers to track their cargo in real-time. It provides the detailed history of an order's status and allows customers to request an agent.
6. **Controller:** The base controller from which all other controllers inherit.
7. **Auth Controllers:** Laravel's built-in authentication controllers (LoginController, RegisterController, etc.) handle secure access to the platform.

## Viva Questions & Answers (Super Detailed)

**Q1: What framework did you use to build this project and why?**
**Answer:** I used Laravel (PHP). Laravel provides an expressive and elegant syntax, built-in features like authentication, routing, and an intuitive ORM (Eloquent). It significantly speeds up development and enforces MVC (Model-View-Controller) architecture, making the codebase clean and maintainable.

**Q2: How does the application handle different types of users?**
**Answer:** The application uses Role-Based Access Control (RBAC). The `User` model has a `role` attribute which can be 'admin', 'agent', or 'customer'. Middleware is used in `routes/web.php` to protect routes based on these roles, ensuring that, for example, a customer cannot access the admin dashboard.

**Q3: You are using MongoDB instead of a traditional SQL database like MySQL. Why?**
**Answer:** MongoDB is a NoSQL, document-oriented database. It allows for flexible schema design, which is highly beneficial for a logistics application where the details of a shipment or tracking events might vary wildly and evolve over time. We used the `mongodb/laravel-mongodb` package to integrate it seamlessly with Eloquent.

**Q4: How does the real-time tracking work for the customer?**
**Answer:** The `OrderStatusUpdate` model records every change in an order's state. When an Agent updates a delivery status, it creates a new update record. The Customer dashboard queries these updates via the `TrackingController` and displays them chronologically, providing a comprehensive timeline.

**Q5: What UI/CSS framework did you use and how did you approach the design?**
**Answer:** I used a combination of Bootstrap 5 for robust grid layouts and standard components, alongside custom CSS (and some Tailwind utilities) for a unique, humanized aesthetic. We focused on a sleek, "gray scale" minimalist design to ensure readability, accessibility, and a modern feel, replacing rigid corporate language with friendly, conversational text.

**Q6: How do you handle security, particularly regarding who can update an order?**
**Answer:** Security is handled via route middleware and controller-level checks. Only users with the 'agent' role who are explicitly assigned to a specific order are permitted to submit status updates to the `AgentController`. Authentication is handled via Laravel's built-in, secure session management.

**Q7: Explain the relationship between the Order and OrderStatusUpdate models.**
**Answer:** It is a One-to-Many relationship. One `Order` has many `OrderStatusUpdate` records. In the `Order` model, there is an `updates()` method defining this relationship, allowing us to easily retrieve the entire history of a specific shipment using Eloquent (e.g., `$order->updates()->get()`).

**Q8: If you were to scale this application, what features would you add next?**
**Answer:** To scale, I would integrate real-time WebSockets (using Laravel Reverb or Pusher) so the customer's tracking page updates instantly without refreshing. I would also add GPS integration for agents to provide live map-based tracking, and implement caching (like Redis) for frequently accessed data such as global network stats.
